## 1. 每支 `.cpp` 的完整結構（依序）

```cpp
// =============================================================================
//  <檔名>  —  <一句話主題>
// =============================================================================
//
// 【主題資訊 Information】
//   簽名／語法、標準版本、時間或空間複雜度、需要的標頭檔
//
// 【詳細解釋 Explanation】        ← 教科書式，這是主體，不是複述 API
//   【1. <子題>】…               分點講「為什麼這樣設計」「解決什麼問題」
//   【2. <子題>】…               把背後的機制講清楚
//
// 【概念補充 Concept Deep Dive】   ← 底層原理／記憶體佈局／編譯器實際做了什麼
//
// 【注意事項 Pay Attention】       ← 陷阱、UB、實作定義、版本差異
//
// ═══════════════════════════════════════════════════════════════════════════
// 【面試題】<主題>
// ───────────────────────────────────────────────────────────────────────────
// 🔥 Q1. <高頻題目>
//     答：<2–4 行>
//     追問：<面試官接著最常追問的那一句，附答>
//
// 🔥 Q2. <另一高頻題>
//     答：…
//
// ⚠️ 陷阱. <看起來會答對、實際答錯的>
//     答：…
//     為什麼會錯：<多數人腦中的錯誤模型是什麼>
// ═══════════════════════════════════════════════════════════════════════════

#include <...>

// 【LeetCode 實戰範例 N】LeetCode ###. <英文原題名>
//   題目：<一句話>
//   為什麼用到本主題：<關聯>
<可編譯、可執行的解法函式>

// 【日常實務範例 N】<真實情境：解析 HTTP header／讀設定檔／log 分級／資料清洗／快取…>
<可編譯、可執行的實務函式>

int main() {
    // 分段輸出，每段前印   std::cout << "=== <段名> ===\n";
}

// 編譯: g++ -std=c++17 -Wall -Wextra <檔名>.cpp -o <輸出名>
//       （多執行緒要加 -pthread；parallel STL 要加 -ltbb 且放在來源檔之後）

// === 預期輸出 ===
// <實際編譯執行取得的輸出，不是手寫的>
```

**四個偵測標記**（用來判斷一支檔案「已加工到哪一步」，也是增量掃描的依據）：

| 加工 | 標記 | 說明 |
|---|---|---|
| A. 面試題 | 檔案含 `【面試題】`（行首 `// 【面試題】`） | 每檔 2–5 題 |
| B. 編譯指令 | 檔案含 `// 編譯:`（Python 為 `# 執行:`） | 該檔**真正需要**的那條 |
| C. 預期輸出 | 檔案含 `=== 預期輸出` | **實跑取得** |

一支「完全做完」的檔案 = A、B、C 三者各出現**恰好一次**。

## 2. 分寸拿捏（不要一律灌到 300 行）

- **`summary.cpp`**：完整規格全上，它是該課的教科書主檔。
- **單一概念的小示範檔（原本 20~60 行）**：詳細解釋可精簡，
  但 **【面試題】必須有**；LeetCode／實務**擇一或從缺**，寧缺勿濫。
- **LeetCode 範例不是每檔都要**：只有真的用得上本主題才加。硬湊一題不相關的比沒有更糟。
  加就要寫**真實存在的題號 + 英文原題名**；**用不上就明白寫一行理由說明為何從缺**
  （例：「拷貝／移動語義是編譯期／資源管理主題，LeetCode 不評測此面向，強加會誤導」）。
- **實務範例要像真的**（解析 log／設定檔／HTTP header／連線池／快取命中率／封包標頭），
  不要寫 `foo/bar` 這種沒情境的東西。

**允許的 LeetCode 題庫**（對得上才用；用計數等可觀察量當輸出證據最好）：
- 設計題：146 LRU Cache、155 Min Stack、705 Design HashSet、707 Design Linked List、
  1603 Design Parking System、1656 Design an Ordered Stream
- 併發題：1114 Print in Order、1115 Print FooBar Alternately、1116 Print Zero Even Odd、
  1117 Building H2O、1195 Fizz Buzz Multithreaded
- 陣列／字串／其他：20、26、27、66、88、118、179、191、204、217、239、283、344、349、1480
- 不在清單但**真的**吻合的題也可以，但要確認題號與英文名正確。

## 3. 硬性要求（每一條都是踩過坑換來的）

1. **一律繁體中文**，技術名詞保留英文；保留每支檔案原本的編碼與換行（BOM/UTF-8/LF 不可亂動）。
2. **每個檔案都要能編譯**（`g++ -Wall -Wextra`，多執行緒 `-pthread`）。
   唯一例外：本來就是「錯誤示範」的檔案 —— 那種要在檔頭明講它為什麼不能編／會崩潰。
3. **UB 不可以描述成固定結果**。不可寫「一定會 crash」「一定印出 X」。
   要區分四種情況：
   - 真正的 UB（未定義行為，結果不保證）
   - 標準保證的 `std::terminate()`（例如 joinable thread 解構）——這種「一定會」是對的
   - 保證阻塞（deadlock；不是 UB，是「一定卡住」）
   - 時序相依（每次可能不同，但不是 UB）
4. **實作定義的數值必須標明**（不是標準保證）。本機（Ubuntu 26.04 / g++ 15.2 / libstdc++ / x86-64）實測：
   - `vector` 成長 **2×**（MSVC 是 1.5×）；`reserve(n)` 剛好配 n；`resize` 用 `max(2*size, n)`
   - `sizeof(std::vector<int>)`=24、`sizeof(std::mutex)`=40、`sizeof(std::mt19937)`=5000
   - list node 24 B、forward_list node 16 B、deque chunk 512 B（128 個 int）
   - libstdc++ SSO 容量 **15**；`hardware_concurrency()`=16
   - `unique_ptr`=8 B、`shared_ptr`=16 B（多一個控制區塊指標）
5. **標準版本要精確**：generic lambda=C++14、`if constexpr`／CTAD／fold expression=C++17、
   `std::erase_if`／`list::unique` 回傳 `size_type`／`namespace a::inline b`=C++20、
   `std::span::at()`=C++26。**不確定就用 `-pedantic-errors` 編來驗證**
   （**只用 `-fsyntax-only` 會被 GCC 當擴充放行，看起來像舊標準也支援**——這是陷阱）。
6. **預期輸出必須是實跑取得**，不可手寫。程式若會崩潰，收集時用 `stdbuf -o0`
   （否則緩衝中的輸出會在 abort 時遺失，實測差 2 行）。
7. **非決定性內容要處理，不要假裝可重現**：
   - 位址：**不要印原始位址**（ASLR 每次不同），改印指標比較的布林值（「是否相同」）。
   - 執行緒 id／耗時：加「每次執行都不同」但書，或**改用計數**（見下）。
   - moved-from 的 `std::string` 內容：**不要印**（valid but unspecified），只印你明確歸零的成員。
   - 未初始化值、無效指標：**連比較都不要**（用無效指標值本身是實作定義，[basic.stc]）——
     dangling／double-free 那幾行保持註解掉。
8. **能用計數就別用計時**：計數可重現、又是更好的證據。
   例：vector 重配置次數 4095 vs 7,998,000（對應 2¹²−1 與 3999·4000/2）、
   copy/move 建構次數、erase-remove 的搬移次數。**計時一律送 stderr，讓 stdout 位元組穩定。**
9. **但書要放在 `// === 預期輸出 ===` 標記的【上方】**，不要放標記下方
   （驗證工具會把標記下方的行當成「預期輸出」逐行比對）。
10. **不要動任何 `Codex` 開頭的路徑。**

## 4. 這門課特別容易踩的真 bug（動手前先自我提醒）

- **成員初始化順序**：初始化列表順序**必須跟宣告順序一致**（`-Wreorder`）。
  本課出現過真正的 heap-buffer-overflow：`char* m_data;` 宣告在 `std::size_t m_len;` **之前**，
  卻寫 `: m_len(strlen(str)), m_data(new char[m_len+1])` → 用未初始化的 `m_len` 配置。
  **編譯一律加 `-Wreorder`，有警告就修。**
- **`<cctype>` 前置條件**：`isspace/isalpha/tolower…` 的引數**必須先轉 `unsigned char`**。
  plain `char` 在 x86-64 是有號的，傳入 UTF-8 的任何一個 byte（0x80~0xFF）會變負值 → UB。
- **`high_resolution_clock`**：在 libstdc++ 上別名是 **`system_clock`**（`is_steady=false`），
  **不是**常說的 `steady_clock`。量測時間要用 `steady_clock`。
- **教科書檔常把同一段程式碼出現兩次**——一次在 `/* */` 說明區塊裡、一次是真程式。
  自動處理時用**最後一次出現**當錨點，並確認錨點在區塊註解之後；任何 helper 要**只追加、不覆寫**
  （曾把內容誤植進說明區的那份 copy）。
- **copy elision / NRVO**：C++17 只保證 **prvalue 初始化**的 elision；NRVO（回傳具名區域變數）
  仍是選擇性的。by-value **參數**不適用 NRVO。要宣稱「這裡少了一次拷貝」前，用
  `-fno-elide-constructors` + C++14 vs C++17 對照驗證。
- **原始碼自己的註解可能是錯的**：本輪抓到多則。看到可疑宣稱要先驗證再照抄，錯的就更正。
- **文件宣稱的輸出可能根本不會出現**（例如某物件從未被 odr-use，`thread_local` 建構沒觸發）——
  **一律實跑確認**。

## 5. 交付前的自我驗證（每支都要做）

1. 用該檔需要的標準編：`g++ -std=c++17 -Wall -Wextra [-pthread] <file> -o /tmp/x`（`-Wreorder` 建議常開）。
2. 在**獨立暫存目錄**真的執行一次（有些範例會寫檔如 `raii_demo.txt`，會污染 repo）。
3. 把實際輸出貼進 `=== 預期輸出 ===` 區塊；**多跑 3~5 次**確認穩定，非決定性的加但書或改計數。
4. 別只比第一行：節錄區塊本來就可能從中段開始；要確認貼的每一行都真的出現在實跑結果。
5. **不可把收集時的暫存路徑寫進檔案**（`/tmp/tmp.XXXX`），要換成 `<執行目錄>`。

## 6. 大量檔案時的分批作業法（避免重工與併發損害）

上一輪 1300 支是用多個背景子代理平行做的。要點：

- **獨占分派**：先算出「待辦清單」，切成互不重疊的批次（`comm -12` 驗證交集為 0），一批一個代理。
- **三態閘門**（每個代理動任何檔案前都要查兩個標記）：
  ```bash
  A=$(grep -cE '^// 【面試題】' "$f"); B=$(grep -c '=== 預期輸出' "$f")
  ```
  - `A=0`：完整增修。
  - `A>=1, B=0`：半成品（前一個代理被砍在中途）→ **只補**缺的 `// 編譯:` + `=== 預期輸出` 區塊，
    **不重加面試題、不重構既有註解**。
  - `A>=1, B>=1`：**已完成 → 完全跳過、不得附加**（附加會造成重複區塊）。
  - append helper 要**硬拒**在已有區塊時再寫。
- **逐檔完成再開下一支**（註解→面試題→LeetCode/實務→編譯→執行→補尾端區塊）。
  這樣萬一被 spend limit／API error 砍掉，最多只損失一支，而不是留下一堆半成品。
- **代理不要再生子代理**（會觸發 API rate limiting；上一輪 7 個代理各自再分裂造成 ~13 個被砍）。
- **代理完成通知可能遲到**：重新分批前要以「當下實際掃描」為準，別用舊快照，否則會把
  已完成的檔案誤併進新批次 → 重複區塊。發現在跑的代理有此風險時，即時發修正訊息。

**commit 一律帶 pathspec，`-F <訊息檔>` 要放在 `--` 之前**：

```bash
git fetch origin                        # 若 Codex 有新 push 先 rebase
git add -- "Claude C++面向對象" "Claude C++ STL課程" "Claude C++ STL多線程課程" \
           "Claude C++11 到 C++26 課程" "Claude Python課程" "Claude Code C++by Examples" \
           "CLAUDE.md" "教材增強_作業規格與模板.md"
# 暫存後複核：git diff --cached --name-only | grep -c '^Codex'   # 必須 0
git commit -F /tmp/commit_msg.txt -- <上面同一組路徑>
git show --name-only --format= HEAD | grep -c '^Codex'          # 必須 0
git push origin main
# 驗證 push 真的成功：git fetch 後比對 HEAD 與 origin/main 的 SHA 一致
```

`git add -A` 或無範圍 `git commit` 會把 Codex **正在 staged 的**工作一起提交——
